# PodMate Architecture — 长期演进路线

## 现在：纯 CLI 工具

```
┌─ 用户 ─┐
│ CLI     │ → Python 模块（转写/翻译/配音/存储）→ 本地文件 + SQLite
└─────────┘
```

所有逻辑在本机运行，用户通过终端操作。

## 阶段 1：API 化（当前正在做）

```
┌─ CLI ─┐ → REST API（本地或远端）→ Python 业务逻辑
└───────┘
```

核心逻辑从 CLI 中剥离，暴露为 REST API。CLI 只做参数解析 + API 调用。
- `podmate/api/` — API 路由层
- `podmate/core/` — 纯业务逻辑（无 typer/console 依赖）
- 部署：初期本地运行，后期上云

## 阶段 2：Web 端

```
┌─ 用户 ─┐                ┌─ CLI ─┐
│ Web UI │  → REST API →  │ API  │  → Python 业务逻辑 → 存储
└────────┘                └───────┘
┌─ Admin ─┐
│ Web UI  │
└─────────┘
```

前端（Next.js / React）通过 API 与后端交互。CLI 和 Web UI 共享同一套 API。

**用户侧功能：**
- 浏览播客、订阅管理
- 播放音频、查看转写/翻译
- 个人设置

**管理员侧功能：**
- 全局播客库管理
- 用户管理
- 系统状态监控（backlog 日志、API 用量、错误率）

## 阶段 3：手机端

```
┌─ iOS App ─┐
│ Android   │ → REST API → API → 业务逻辑 → 存储
│ 小程序    │
└───────────┘
```

手机端（React Native / Taro 跨端方案）复用同一套 API。
- 核心：播客播放 + 转写/翻译阅读
- 离线：下载音频和文本离线使用

## 架构原则

1. **分层分离**：CLI 层（展示）与业务逻辑层（核心）严格分离，不交叉依赖
2. **接口先行**：每个关键操作都有清晰的 API 契约，CLI 只是客户端之一
3. **可观测性**：所有 API 调用记录 backlog 日志，Web/App 端可复用
4. **渐进式**：不做超前设计，每次只为一个阶段做架构准备

## 当前阶段约束

- 不引入 Web 框架、不做多用户、不改 DB schema
- 新增功能（translate 命令、backlog 日志）考虑接口可复用性
- 配置系统、数据目录结构预留 scope 化能力
